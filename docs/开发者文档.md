# 开发者文档

## 1. 技术栈
- 前端：Vue 3 + Vite
- 后端：Spring Boot 3.2.x、MyBatis-Plus、Redis、MySQL、WebSocket、Springdoc OpenAPI
- AI 服务：FastAPI、Ultralytics YOLO、OpenCV、Kornia/LoFTR、PyTorch
- 模型训练：PyTorch（disease_matching 目录）

## 2. 关键业务链路（从上传到展示）
1. 无人机上传图片：后端 `POST /api/v1/drones/upload`
2. 后端保存原图并初始化入库（inspection_image）
3. 后端异步调用 AI 检测接口（/predict）
4. AI 返回检测框与结果图路径，后端写入 defect_detail，并更新 inspection_image
5. 相邻帧可能触发 SIFT 单应矩阵做局部合并（减少重复）
6. 任务结束时触发屏障：收到结束信号 + 待处理图片为 0 -> 触发全局匹配
7. 全局匹配：GNN / LoFTR 兜底，最终形成 defect_entity 与 detail 关联
8. 地图瓦片查询：/api/v1/map/tile，从数据库查询实体并结合 Redis 缓存加速
9. 前端地图展示与详情弹窗：点击点位拉取详情列表并展示图片链接

## 3. 事件与异步机制
- AI 推理异步执行：DroneAsyncService（@Async）
- AI 结果事件：AiResultEvent -> DroneService.onAiResult（写库/失败标记）
- 病害告警事件：DefectDetectedEvent -> WebSocketAlertListener 推送
- 逆地理编码补全：RoadInfoUpdateEvent -> RoadInfoListener 异步补全 roadName/address
- 任务结束屏障：TaskStateManager 在“结束信号 + pendingImages=0”时触发 GlobalMatchService

说明：
- 该设计避免了“任务已结束但图片仍在处理”的竞态，保证全局匹配只在数据齐全时进行。

## 4. 前端关键交互
- 航线点选：TaskManager 通过自定义事件与地图页交互（route-picking-toggle / route-point-picked 等）
- 瓦片加载：地图根据当前视口计算 z/x/y 并请求 /api/v1/map/tile
- WebSocket：地图页/采集页均可接收遥测消息，并绘制无人机轨迹
